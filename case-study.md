# コーチング事業向けAI実行基盤と管理ダッシュボード

## 概要

子どもの目標達成を支援する1on1コーチング事業向けに、Discord Bot と Webアプリを中心とした運用基盤を設計・実装した。  
対象は、日々の振り返り対話、面談後の週次レポート作成、保護者向け月次レポート作成、承認付き配信、生徒情報の管理である。  
要件整理、仕様策定、データ設計、Bot 実装、管理画面実装、既存データ移行、デプロイ、導入後の保守までを単独で担当した。

初期検証を含めたプロジェクト全体は約3か月、方向性確定後の本開発は約1.5か月で行った。  
現在も継続運用されている。

## 事業と運営状況

クライアントは、週1回の面談と日々の振り返り記録を組み合わせて、子どもの目標達成を支援していた。  
生徒数は約50名で、運営は実質的に1人のコーチに依存していた。

40名程度までは無理なく回せても、50名を超えると、面談後の記録整理、保護者向けレポート作成、提出状況の確認、面談前の情報整理に使う時間が増え、運営改善や拡張に使える時間が減っていた。  
過去には複数コーチ体制も試していたが、情報共有と運営管理が機能せず、数か月で1人体制に戻っていた。

## 導入前の業務

導入前は、支援そのものとは別に、記録とレポート作成の負荷が継続的に発生していた。

- 面談後の週次レポート: 生徒1人あたり毎週約3分
- 保護者向け月次レポート: 生徒1人あたり作成から送信まで約5分

生徒数が約50名の運用では、週次レポートだけで約150分/週、月次レポートで約250分/月の作業が発生していた。  
これに加えて、議事録の手入力、提出状況の確認、過去記録の参照、送信対応、面談準備が個別に発生していた。

この構造では、コーチの時間が面談そのものではなく、面談後の整理、保護者向け報告、状態把握のための確認作業に取られやすい。  
その結果、支援内容の改善、複数コーチ運営への移行、属人的な知見の整理といった中長期の整備が後回しになっていた。

## 課題

整理すると、課題は次の4点だった。

1. 面談後の記録とレポート作成に継続的な工数がかかっていた
2. 生徒情報が分散しており、経過と現状を追いにくかった
3. 保護者向けレポートの品質と送信管理が手作業に依存していた
4. 情報管理と運営導線が1人のコーチに依存しており、複数人で運営しにくかった

この4点は独立した問題ではなく、互いに影響していた。  
たとえば、情報が分散しているため面談準備に時間がかかり、その結果としてレポート作成や確認作業が後ろ倒しになり、さらにコーチの時間が圧迫される。  
この状態では、支援品質の改善も、事業の拡張も進めにくい。

## この課題を優先した理由

この案件で優先度が高かったのは、単純な工数削減ではなく、運営構造の改善だった。

事業自体は直ちに赤字化するような構造ではなかったが、支援品質と収益がコーチ個人の稼働量に依存している以上、長期的には利益の拡大も休暇の確保も難しい。  
また、過去に複数コーチ体制が継続できなかったことから、人を増やせば解決する問題でもなかった。

したがって、この案件では、レポートを自動生成することそのものではなく、記録、要約、承認、送信、参照を一つの流れとして整理し、複数人で追える状態を作ることを優先した。

## 制約条件

設計上の主な制約は次の通りだった。

- 年明けから運用開始する前提で、方向性確定後の開発期間は約1.5か月
- 月1万円以内の運用コスト
- 未成年を含む利用者を扱うため、権限制御と個人情報保護が必要
- 保護者向けレポートは対外送信物であり、誤送信と誤内容を避ける必要がある
- 既存運用を止めずに移行する必要がある
- 利用者の年齢が小学校低学年から18歳まで広く、認知負荷を上げにくい

この制約は、そのまま設計判断に反映している。  
短期間導入が必要だったため、インフラは過剰構成にせず、認証・DB・ストレージをまとめて扱える構成を優先した。  
コスト制約があったため、月次レポートの生成方法やモデル選定では token 使用量と品質の両方を比較した。  
未成年と対外送信物を扱うため、自動化の範囲は広げつつも、誤りが事故に直結しない状態遷移と承認フローを入れた。

## 担当範囲

この案件では、要件整理から仕様策定、実装、導入、運用開始後の保守までを一貫して担当した。

担当範囲は次の通り。

- 要件整理と仕様策定
- Discord Bot の導線設計
- AIコーチの対話設計
- Webアプリの情報設計と UI / UX 設計
- データベース設計
- 既存データの移行
- 週次レポート生成フローの設計
- 月次レポート生成フローの設計
- 編集 / 承認 / 送信フローの設計
- 保護者向けLINE送信連携
- RAG を用いた過去情報検索の設計
- デプロイ構成の設計と本番投入
- 生徒向け導入資料、説明文、動画、マニュアルの整備
- リリース後の保守、エラー対応、機能追加

クライアント側には課題意識と実現したい方向性はあったが、何を先に整備するか、どの業務をシステムに載せるか、どこまでを自動化し、どこに人の確認を残すかは自分が整理した。

## 比較した案

### 生徒側の導線

候補は LINE、独自Webアプリ、独自モバイルアプリ、Discord だった。  
比較軸は、継続利用のしやすさ、導入速度、権限制御、運用側通知の扱いやすさだった。

独自アプリは柔軟だが、今回の期間制約には合わない。  
LINE は日常導線としては強いが、今回必要だったプライベートチャンネル設計や運用側の通知導線との相性では Discord の方が扱いやすかった。  
このため、Bot と管理導線を同じ基盤上に置きやすい Discord を採用した。

### 管理方法

候補はスプレッドシート、Notion、独自管理画面だった。  
比較軸は、情報の横断参照、運営判断への再利用、複数人運営時の追いやすさだった。

スプレッドシートや Notion は記録を残す用途には向いているが、生徒ごとの履歴、提出状況、議事録、送信状況、検索結果を一つの文脈で追うには不足があった。  
また、過去に複数コーチ体制が続かなかった背景にも情報管理の弱さがあった。  
このため、記録保存よりも状態把握と運営判断を優先し、独自の管理画面を採用した。

### レポート生成方式

候補は、毎回すべての会話履歴から生成する方式と、途中で要約を作って再利用する方式だった。  
比較軸は、実装の単純さ、品質、token 使用量、長期運用時の安定性だった。

毎回全履歴を使う方式は単純だが、履歴が増えるほど入力 token が増え、コストの見通しが悪くなる。  
月次レポートの入力元を「その月の全会話」から「週次レポート群」に切り替えると、月次工程の入力 token は 756,000 から 77,143 に下がり、月次工程では約89.8%、システム全体では約28.2%削減できる試算だった。  

このため、日々の記録から週次レポートを作り、その週次レポート群から月次レポートを作る構成を採用した。  
その際、再要約による情報欠落を防ぐため、週次レポートは自由文ではなく固定項目を持たせた。  
週次レポートの長さは 300〜500字程度を前提にした。

### 保護者向け送信

候補は完全自動送信と承認付き送信だった。  
比較軸は、工数、誤送信時のリスク、文面品質の調整余地だった。

対外送信物である以上、誤送信や誤内容を防ぐ必要があったため、生成後にコーチが確認できる承認付き送信を採用した。

### 検索方式

候補は単純なナレッジベース運用と RAG を使った検索だった。  
比較軸は、過去事例の再利用、意味的に近い記録の抽出、現在の支援への当てはめやすさだった。

単純なナレッジベースでも記録の保存はできるが、今回必要だったのは「似た状況の生徒が過去にどのような行動を取り、どのような変化があったか」を取り出すことだった。  
このため、検索は RAG 前提で設計した。

## 採用した構成

運用フローは次の通り。

1. 生徒が Discord 上で日々の振り返りを入力する
2. AIコーチが対話を返す
3. 記録を保存する
4. 1週間分の記録から面談後の週次レポートを生成する
5. 1か月分の週次レポートから保護者向け月次レポートを生成する
6. 月次レポートをコーチ専用チャネルに送る
7. コーチが編集または承認する
8. 承認後に保護者へ送信する
9. 管理画面で生徒ごとの履歴、提出状況、レポート履歴、送信状況、検索結果を参照する

この構成にした理由は、日々の入力、面談後の整理、保護者向け送信、生徒情報の参照がそれぞれ別の作業になっていたためである。  
それらを一つの運用フローとして接続することで、記録を使い回せる状態にし、同じ情報を何度も別の場所で探したり書き直したりする必要を減らした。

## 設計上の判断

### 週次レポートを中間成果物として扱う

週次レポートは面談後の共有資料であると同時に、月次レポート生成の入力でもある。  
このため、単なる要約ではなく、月次生成で必要になる情報を残す構造にした。

### 完全自動送信ではなく、承認付き送信にする

保護者向けレポートは対外送信物であるため、内容確認を残した。  
自動化は維持しつつ、品質責任は人が持てるようにした。

### Bot と管理画面を分ける

日々の対話は Bot 側、履歴確認と運営判断は管理画面側に分けた。  
チャット導線と情報参照導線は役割が異なるためである。

### 年齢ごとに対話レベルを変える

対象年齢が広いため、同一の対話では成立しにくい。  
インターフェースは単純に保ち、対話の深さと語彙はAI側で調整する形にした。

### モデルを用途別に分ける

日次対話、保護者向けレポート、検索補助では、必要な品質とコストが異なる。  
そのため、用途ごとにモデルを分けた。

### インフラは短期導入と保守性を優先する

- Supabase: 認証・DB・ストレージを短期間で立ち上げるため
- Render: デプロイと運用負荷を抑えるため
- Docker: ローカルと本番の差分を小さくし、保守しやすくするため

## 実装上の論点

### 情報を圧縮しながら品質を保つこと

段階的要約は token 使用量に効くが、要約を重ねると情報が落ちる。  
初期段階では品質の不安定さや情報欠落が起きたため、月次レポートで必要になる観点を先に定義し、それを週次レポート側に持たせる形に修正した。

### 自動送信の安全性をコード側で担保すること

保護者向けレポートは対外送信物であるため、AIの出力品質だけに依存しない構成が必要だった。  
次のような処理を入れた。

- 生徒ID、チャンネルID、保護者LINE ID の紐付け
- 承認状態チェック
- 送信対象チェック
- 二重送信防止
- 送信状況チェック
- 送信ジョブの idempotency key
- 編集前 / 承認前 / 送信後の状態遷移管理
- 失敗時の再送制御と再送上限
- webhook 署名検証と callback 検証
- 監査用ログの保存

### データを検索・分析可能な形に整備すること

既存データは Excel 上のカルテ、議事録、過去記録などに分散していた。  
このため、そのままRAGに投入するのではなく、まず生データを整形し、検索対象として使える単位へ再構成した。

また、生ログだけの RAG では、検索語の揺れ、相関と因果の混同、時系列の欠落が起きやすい。  
この用途では、原文検索用の生テキストに加えて、目標、行動、障害、継続度、結果などを構造化したデータを併設し、hybrid search と rerank を前提にした方が扱いやすい。

### 年齢差への対応

対象ユーザーは小学校低学年から18歳まで広かった。  
同じ語彙や問いかけでは成立しないため、年齢ごとにプロンプトを分け、対話の深さと語彙を調整した。

### セキュリティを多層で入れること

未成年と個人情報を扱うため、次の対策を入れた。

- チャンネル分離
- 権限制御
- 入力制御
- ツール呼び出し制限
- プロンプト分離
- 固定 system prefix と user content の厳密分離
- retrieval scope 制限
- モデルに渡す前のデータマスキング
- 管理画面側の閲覧権限分離

## 結果

導入後の結果は次の通り。

- 面談後の週次レポート作成: 実質0分
- 保護者向け月次レポート: 1人あたり約30秒で編集・承認
- 解約率: 導入前後の月次比較で30%低下
- 体制面: 新たに4名のコーチを採用可能
- 事業面: 1か月で生徒数30名増加
- 顧客体験: 日次内省の頻度が増え、目標達成率の向上を確認

事務作業の削減によってコーチ側の時間が生まれ、管理画面によって複数人で追える受け入れ体制が整ったことが、体制拡張に直接つながった。

## 振り返り

有効だった判断は次の3点。

- 日々の記録 → 週次レポート → 月次レポートの階層構造にしたこと
- 生徒側の導線を Discord に寄せたこと
- Bot だけで閉じず、管理画面を独立させたこと

今後の改善点は次の3点。

- 生徒ごとの会話履歴を構造化して保存し、その子専用の長期コンテキストを持つこと
- 対話品質を継続的に評価する基盤を作ること
- 管理画面の分析機能を拡張すること

将来的には、AIメモリー層の導入、面談そのものの一部自動化、コーチ固有の知見の構造化と再利用まで進める余地がある。
